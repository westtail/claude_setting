# テスト要件

## テストカバレッジ目標: 80%

## テストの種類

1. **ユニットテスト** - 個別の関数、モジュール、クラス
2. **統合テスト** - 複数のモジュール間の連携、外部サービスとの接続

## テスト駆動開発（TDD）

推奨ワークフロー：

1. テストを最初に作成（RED）
2. テストを実行 - 失敗すること
3. 最小限の実装を作成（GREEN）
4. テストを実行 - 合格すること
5. リファクタリング（REFACTOR）
6. カバレッジを検証

## テスト設計の原則

- **独立性**: 各テストは他のテストに依存しない
- **再現性**: 何度実行しても同じ結果
- **明確性**: テスト名で何をテストしているか分かる
- **単一責任**: 1つのテストで1つの振る舞いを検証

## テストの構造（AAA パターン）

1. **Arrange** - テストの準備（データ、モックの設定）
2. **Act** - テスト対象の実行
3. **Assert** - 結果の検証

## テスト失敗時の対応

1. エラーメッセージを確認
2. テストの分離を確認（他のテストの影響がないか）
3. モック・スタブが正しいか検証
4. 原則：テストではなく実装を修正（テスト自体が間違っている場合を除く）

## 良いテストの特徴

- 実行が高速
- 失敗時に原因が特定しやすい
- メンテナンスが容易
- 実装の詳細ではなく振る舞いをテスト
