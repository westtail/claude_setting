# コーディングスタイル

## 命名規則

明確で意図が伝わる名前を使用：

- 変数・関数名は目的を表す（`x`ではなく`userCount`）
- 略語は避け、検索可能な名前を使用
- ブール値は`is`/`has`/`can`で始める
- 関数名は動詞で始める（`getUser`、`calculateTotal`）
- 定数は大文字スネークケース（`MAX_RETRY_COUNT`）

## 関数設計

### 単一責任の原則

- 1つの関数は1つのことだけを行う
- 関数は小さく保つ（目安：20-30行以内）
- 引数は3つ以下が理想（多い場合はオブジェクトにまとめる）

### 副作用を最小化

- 可能な限り純粋関数を使用
- 入力を変更せず、新しい値を返す
- グローバル状態への依存を避ける

### 早期リターン

深いネストを避け、ガード節で早期に抜ける：

- 異常系を先に処理してリターン
- 正常系の処理をメインフローに

## ファイル・モジュール構成

- 高凝集、低結合を目指す
- 1ファイルあたり200-400行を目安（最大800行）
- 機能/ドメインでファイルを整理
- 循環依存を避ける

## エラーハンドリング

- エラーは適切なレイヤーで処理
- ユーザー向けには分かりやすいメッセージ
- 開発者向けには詳細なログ
- 回復可能なエラーと致命的なエラーを区別

## 入力バリデーション

- システム境界（外部入力）では必ず検証
- 早い段階でバリデーションを行う（Fail Fast）
- エラーメッセージは具体的に

## コードの可読性

### コメント

- 「なぜ」を説明し、「何を」は説明しない
- 自明なコードにはコメント不要
- 複雑なビジネスロジックには説明を追加

### フォーマット

- 一貫したインデントとスタイル
- 空行で論理的なブロックを分離
- 1行の長さは80-120文字を目安

## 設計原則

### DRY（Don't Repeat Yourself）

- 同じロジックを複数箇所に書かない
- ただし、早すぎる抽象化は避ける

### KISS（Keep It Simple, Stupid）

- 必要以上に複雑にしない
- 将来の要件を推測して過剰設計しない

### YAGNI（You Aren't Gonna Need It）

- 今必要ないものは作らない
- 実際に必要になってから実装

## コード品質チェックリスト

作業完了前に確認：

- [ ] 命名が明確で意図が伝わる
- [ ] 関数が小さく、単一責任
- [ ] ネストが深すぎない（4レベル以下）
- [ ] エラーが適切に処理されている
- [ ] デバッグ用コードが残っていない
- [ ] マジックナンバーがない
- [ ] コードが自己文書化されている
