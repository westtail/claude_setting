# テスト駆動開発ワークフロー

このスキルは、すべてのコード開発が包括的なテストカバレッジを持つTDD原則に従うことを保証。言語やフレームワークに依存しない汎用的なガイドライン。

## 起動タイミング

- 新機能や機能の作成時
- バグや問題の修正時
- 既存コードのリファクタリング時
- APIエンドポイントの追加時
- 新しいコンポーネントやモジュールの作成時

## 基本原則

### 1. コードより先にテスト（RED-GREEN-REFACTOR）

```
RED     → まず失敗するテストを書く
GREEN   → テストを通過させる最小限のコードを書く
REFACTOR → テストをグリーンに保ちながらコードを改善
REPEAT  → 次の要件に進む
```

### 2. カバレッジ要件

- 最低80%カバレッジ（ユニット + 統合 + E2E）
- すべてのエッジケースをカバー
- エラーシナリオをテスト
- 境界条件を検証

### 3. テストタイプ

#### ユニットテスト

- 個々の関数とユーティリティ
- クラスやモジュールのロジック
- 純粋関数
- ヘルパーとユーティリティ

#### 統合テスト

- APIエンドポイント
- データベース操作
- サービス間のやり取り
- 外部API呼び出し

#### E2Eテスト

- 重要なユーザーフロー
- 完全なワークフロー
- UIの自動化
- システム全体の動作確認

## TDDワークフローステップ

### ステップ1：ユーザーストーリーを書く

```
[役割]として、[アクション]したい、それによって[メリット]を得たい

例：
ユーザーとして、商品をセマンティックに検索したい、
それによってキーワードが正確でなくても関連する商品を見つけられる。
```

### ステップ2：テストケースを設計

各ユーザーストーリーに対して包括的なテストケースを作成：

```
テストスイート: セマンティック検索

  テスト: クエリに関連する商品を返す
    準備: テスト用商品データを用意
    実行: 検索関数を呼び出す
    検証: 関連する商品が返されること

  テスト: 空のクエリを適切に処理する
    準備: 空のクエリ文字列
    実行: 検索関数を呼び出す
    検証: 空の結果またはエラーが返されること

  テスト: キャッシュが利用不可の場合はフォールバック
    準備: キャッシュを無効化
    実行: 検索関数を呼び出す
    検証: 代替検索が実行されること

  テスト: 結果を類似度スコアでソートする
    準備: 複数の類似商品データ
    実行: 検索関数を呼び出す
    検証: スコア降順でソートされていること
```

### ステップ3：テストを実行（失敗するはず）

```bash
# 言語に応じたテストコマンドを実行
# 例: pytest, go test, npm test, rspec, etc.

# テストは失敗するはず - まだ実装していない
```

### ステップ4：コードを実装

テストを通過させる最小限のコードを書く：

```
# 擬似コード
function searchProducts(query):
    # テストを通過させる最小限の実装
    # 過剰な実装を避ける
    return results
```

### ステップ5：再度テストを実行

```bash
# テストを実行
# テストは通過するはず
```

### ステップ6：リファクタリング

テストをグリーンに保ちながらコード品質を改善：

- 重複を削除（DRY原則）
- 命名を改善
- パフォーマンスを最適化
- 可読性を向上
- 設計パターンを適用

### ステップ7：カバレッジを検証

```bash
# カバレッジレポートを生成
# 80%以上のカバレッジを達成したことを確認
```

## テストパターン

### AAA（Arrange-Act-Assert）パターン

```
テスト: ユーザー作成が正常に動作する

  # Arrange（準備）
  ユーザーデータを準備する
  データベースをクリーンな状態にする

  # Act（実行）
  ユーザー作成関数を呼び出す

  # Assert（検証）
  ユーザーが作成されたことを確認
  IDが割り当てられたことを確認
  タイムスタンプが設定されたことを確認
```

### Given-When-Then パターン（BDD）

```
シナリオ: ユーザーがログインできる

  Given: 有効なユーザーアカウントが存在する
  When: 正しい認証情報でログインする
  Then: アクセストークンが返される
  And: ユーザー情報が取得できる
```

### ユニットテストのパターン

```
テストスイート: Calculator

  テスト: 正の数を加算できる
    入力: 2, 3
    期待: 5

  テスト: 負の数を加算できる
    入力: -1, -2
    期待: -3

  テスト: ゼロ除算でエラー
    入力: 10, 0
    期待: エラー発生
```

### 統合テストのパターン

```
テストスイート: GET /api/users

  テスト: ユーザー一覧を正常に返す
    準備: テストユーザーを作成
    実行: GET /api/users
    検証:
      - ステータス200
      - レスポンスがJSON
      - ユーザー配列を含む

  テスト: 認証なしで401を返す
    準備: 認証ヘッダーなし
    実行: GET /api/users
    検証:
      - ステータス401

  テスト: データベースエラーを適切に処理する
    準備: データベース接続をモック
    実行: GET /api/users
    検証:
      - ステータス500
      - エラーメッセージを含む
```

### E2Eテストのパターン

```
テストスイート: ユーザー登録フロー

  テスト: 新規ユーザーが登録できる
    1. 登録ページにアクセス
    2. フォームに入力:
       - 名前: "テスト太郎"
       - メール: "test@example.com"
       - パスワード: "secure123"
    3. 送信ボタンをクリック
    4. 成功メッセージが表示されることを確認
    5. ダッシュボードにリダイレクトされることを確認

  テスト: 重複メールでエラー
    1. 既存ユーザーのメールで登録を試行
    2. エラーメッセージが表示されることを確認
    3. フォームがリセットされないことを確認
```

## テストファイル構成

## モック・スタブ・フェイク

### モック（呼び出しを検証）

```
テスト: 通知が送信される

  # メール送信サービスをモック
  モック: EmailService.send() → 呼び出しを記録

  # 実行
  ユーザー登録を実行

  # 検証
  EmailService.send()が1回呼ばれたことを確認
  引数にユーザーメールが含まれることを確認
```

### スタブ（固定値を返す）

```
テスト: 外部API障害時のフォールバック

  # 外部APIをスタブ
  スタブ: ExternalAPI.fetch() → エラーを返す

  # 実行
  データ取得を実行

  # 検証
  キャッシュからデータが返されることを確認
```

### フェイク（軽量な代替実装）

```
テスト: データベース操作

  # インメモリデータベースを使用
  フェイク: Database → InMemoryDatabase

  # 実行
  CRUD操作を実行

  # 検証
  データが正しく保存・取得されることを確認
```

## テストダブルの選択基準

| 種類     | 用途                   | 例                   |
| -------- | ---------------------- | -------------------- |
| ダミー   | パラメータを埋めるだけ | 使用されないロガー   |
| スタブ   | 固定の戻り値を提供     | 設定値を返すサービス |
| スパイ   | 呼び出しを記録         | イベント送信の確認   |
| モック   | 期待を検証             | API呼び出しの検証    |
| フェイク | 軽量な実装             | インメモリDB         |

## よくあるテストのミスを避ける

### ❌ 間違い：実装詳細をテスト

```
# 内部状態をテスト（脆い）
コンポーネントの内部変数が5であることを確認
```

### ✅ 正解：動作をテスト

```
# ユーザーに見える結果をテスト（堅牢）
画面に「カウント: 5」が表示されることを確認
```

### ❌ 間違い：脆弱なセレクター

```
# CSSクラスに依存（変更されやすい）
要素 ".btn-primary-xyz" をクリック
```

### ✅ 正解：セマンティックなセレクター

```
# 意味に基づく選択（安定）
"送信" というテキストのボタンをクリック
data-testid="submit-button" をクリック
```

### ❌ 間違い：テスト間の依存

```
テスト1: ユーザーを作成
テスト2: 同じユーザーを更新（テスト1に依存）
```

### ✅ 正解：独立したテスト

```
テスト1: ユーザーを作成
  準備: 独自のテストデータ

テスト2: ユーザーを更新
  準備: 独自のテストデータ（テスト1とは無関係）
```

### ❌ 間違い：非決定的なテスト

```
# 現在時刻に依存（不安定）
結果の日付が今日であることを確認
```

### ✅ 正解：決定的なテスト

```
# 時刻をモック（安定）
システム時刻を固定値に設定
結果の日付が固定値であることを確認
```

## カバレッジ目標

| コードタイプ           | 目標    |
| ---------------------- | ------- |
| 重要なビジネスロジック | 100%    |
| パブリックAPI          | 90%以上 |
| ユーティリティ関数     | 85%以上 |
| 一般コード             | 80%以上 |
| 生成コード・設定       | 除外可  |

## 継続的テスト

### 開発中

```bash
# ウォッチモードでテストを実行
# ファイル変更時に自動で該当テストが実行される
```

### コミット前（プレコミットフック）

```bash

```

### CI/CDパイプライン

```yaml

- 閾値未満ならビルド失敗

- 完全なテストスイートを実行
- E2Eテストを実行
- カバレッジをレポート
```

## テストのベストプラクティス

### 構造

1. **テストを先に書く** - 常にTDD
2. **1テスト1アサート** - 単一の動作に集中
3. **記述的なテスト名** - 何がテストされているかを説明
4. **AAAパターン** - Arrange-Act-Assert で構造化

### 品質

5. **外部依存をモック** - ユニットテストを分離
6. **エッジケースをテスト** - null、空、境界値、大きな値
7. **エラーパスをテスト** - ハッピーパスだけでなく
8. **テストを高速に** - ユニットテストは各50ms未満

### メンテナンス

9. **テスト後はクリーンアップ** - 副作用なし
10. **重複を避ける** - ヘルパー関数・フィクスチャを活用
11. **テストも本番コードと同じ品質** - リファクタリング対象
12. **カバレッジレポートをレビュー** - ギャップを特定

## 成功指標

- ✅ 80%以上のコードカバレッジを達成
- ✅ すべてのテストが合格（グリーン）
- ✅ スキップまたは無効化されたテストなし
- ✅ 高速なテスト実行（ユニットテストは30秒未満）
- ✅ E2Eテストが重要なユーザーフローをカバー
- ✅ テストが本番前にバグをキャッチ
- ✅ 新機能には必ずテストが付随
- ✅ リグレッションテストが既存機能を保護

---

**覚えておく**：テストはオプションではありません。自信を持ったリファクタリング、迅速な開発、本番の信頼性を可能にするセーフティネットです。
